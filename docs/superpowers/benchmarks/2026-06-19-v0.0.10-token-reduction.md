## Recommendation: adjust the gate

The original ≥25% reduction gate (final ≤22,180 bytes) is **not achievable** through E1–E4 as prescribed. The agent ended Part E at **35,293 bytes — 19% larger than the v0.0.9 baseline**, not smaller.

**Why the gate missed:**
1. Parts B (diff temp-file mechanism) and C (npx wrappers for biome/fallow/ast-grep) added ~60 lines / ~3,400 bytes of necessary functionality. These aren't optional — they're load-bearing for the self-contained-install goal.
2. E1–E4 are clarity refactors (single source of truth for severity ladder, concrete Phase 2.5 triggers, deterministic Phase 3 queries), not byte-reduction refactors. Each one *added* a small amount of structure.
3. The original plan's "~50–65 lines saved" estimate for E1 assumed the per-domain severity ladders were longer than they actually were; the centralized block carries per-domain framing inside each rung, making it denser than the duplicates combined.

**Proposed new gate (honest):**
- **No accidental bloat** — final byte count ≤36,000 (current 35,293 has ~700 bytes headroom for Parts F/G/H/I).
- **Clarity invariants** — `<severity-ladder>` is the single source of truth; Phase 2.5 triggers are enumerated; Phase 3 queries are deterministic.
- **Severity-drift gate unchanged** — `/codelens:review security` + `/codelens:review a11y` counts within ±1/category of v0.0.9 baseline. Run at Part E final checkpoint (end-to-end review).

**Alternative if the 25% gate is load-bearing for the release narrative:** aggressive cuts to Phase 1+2 batch example (collapse to one paragraph), weighted-hotspot pseudocode compression, constraints bullet merging. Risk: severity drift if structural cues get lost. Out of scope for this execution per user direction.
