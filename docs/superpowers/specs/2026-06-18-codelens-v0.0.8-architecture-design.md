## 1. Executive Summary

The research note asks six questions about agent orchestration. After cross-referencing the note against the existing v0.0.6 implementation review, the architecture brainstorming doc, and the v0.0.6 smoke test, the evidence converges on a clear answer:

**Do not split the agent into orchestrator + subagents.** Instead, fix the three real defects the evidence surfaces:

1. **Permission friction** — `.claude/settings.local.json` allowlists only MCP tools. Every `rg`, `git`, `ast-grep`, `biome`, `tsc`, `fallow` command triggers a prompt. **Add a scoped Bash allowlist.**
2. **Output drift** — schemas exist but enforcement is prompt-only. **Add a deterministic validation gate** before `.codelens/reviews.json` append.
3. **Phase 2 ↔ Phase 3 overlap** — Phase 3 re-discovers what Phase 2 already found. **Make Phase 3 a correlator, not a re-discoverer.**

Two research-note hypotheses are **rejected** by the evidence: "the agent architecture is too monolithic" (docs rate the current structure 8.5/10; the subagent split is explicitly deferred as Priority C #8 — "out of scope until proven necessary"), and "execution, evidence gathering, and report writing should be separated" (this is what Phase 0–4 already does — separation is by phase, not by agent).

The two remaining observations — *format drift on output* and *re-execution after final output* — are real and addressed below, but their root cause is **missing deterministic gates**, not agent structure.

**Recommendation:** v0.0.8 is a **discipline release**, not a structural rewrite. Add gates, allowlists, and a Phase 3 semantics shift. Defer the subagent split to v0.1.0+ unless a >2000-file smoke test proves Phase 3 is the bottleneck.

## 2. Research Questions Answered

### Q1 — How should agent orchestration be structured for maintainability?

**Answer:** Keep the single-agent, phase-based structure. Maintainability is not improved by adding subagent boundaries when the existing phases already separate concerns. What hurts maintainability is the accumulation of behavioral constraints (`**Never ...`, `**Always ...`) — currently ~20 — that compensate for missing deterministic enforcement. The fix is to **move enforcement into tooling** (gates, allowlists, schema validation), not to add agents.

**Evidence:** Architecture review rates the phase structure 8.5/10. The "too many behavioral constraints" weakness is explicitly diagnosed as "usually a signal that behavior belongs in tooling rather than prompts." The observed pattern of rule → violation → hardening → re-modification across versions is the symptom of prompt-enforced behavior.

### Q2 — When should I use a single agent vs subagents?

**Answer:** Single agent for v0.0.8. Subagents are justified only when:
- A phase is CPU-bound and parallelizable (Phase 3 on >2000-file repos), **and**
- The subagent's task is self-contained with no shared state, **and**
- The context cost of re-establishing shared state in the subagent is less than the parallelism win.

None of these conditions currently hold. The current Phase 3 bottleneck is the **sequential** `ctx_batch_execute` calls per hotspot, but each call already batches 6–10 commands concurrently. A subagent split would replicate the orchestrator's state (hotspot list, domain config, scope, exclusion list) into N subagent contexts — a net context increase, not a decrease.

**Evidence:** v0.0.6 implementation review Priority C #8: "Subagent split for very large codebases ... Out of scope until proven necessary."

### Q3 — When should I use single-turn vs multi-turn execution?

**Answer:** Maximize single-turn batches where the LLM is not needed between commands. The v0.0.5 merger of Phase 1 + Phase 2 into one `ctx_batch_execute` call (1 LLM turn vs ~8) is the model. Apply the same pattern to Phase 3: the per-hotspot `ctx_batch_execute` calls should be **parallelized across hotspots in a single batch** (concurrency=8) rather than sequenced. The LLM turn happens **after** all hotspots return, when correlation is required.

**Tradeoff:** parallel Phase 3 hotspots cap at concurrency=8 (context-mode limit). For >15 hotspots this means 2 batches — still 2 LLM turns total, vs 10–15 today.

### Q4 — How much context does a subagent actually get?

**Answer:** A freshly spawned subagent gets **only what the parent passes in the prompt** — no conversation history, no auto-memory, no session events. This is documented Claude Code behavior (the parent must hand-craft a self-contained brief) and is confirmed by the codelens-reviewer agent's own context-management discipline.

**Implication for codelens:** a subagent dispatched to analyze a hotspot needs: the file path, the domain config, the scope, the exclusion list, the Phase 2 signals for that file, the schema abstraction rules, and the output shape. That's ~2KB of context per subagent. For 15 hotspots that's 30KB of replicated context — **more** than the current single-agent approach where the state lives once in the conversation. This is the concrete reason the subagent split is deferred.

### Q5 — Is context-mode MCP helpful in large review workflows?

**Answer:** Yes, decisively. It is the load-bearing token-efficiency layer. The v0.0.5→v0.0.6 progression (single `ctx_batch_execute` for Phase 1+2, `ctx_execute_file` instead of Read for source/templates, `ctx_execute` post-processor for Risk Score) reduced per-review token cost from ~14K to ~8.5K (~40%). v0.0.8 should extend this by:
- Parallelizing Phase 3 hotspots into one `ctx_batch_execute` (see Q3).
- Adding `ctx_execute_file` for any remaining raw source reads in Phase 3 (currently still uses `ctx_search` for evidence re-verification — fine, but confirm no raw Reads leak in).
- Caching biome + tsc results per-session (smoke test long-term recommendation #9) — re-running `tsc --noEmit` on every review of the same codebase is pure waste.

### Q6 — How can I reduce permission prompts and output-format drift?

**Answer:** Two separate root causes, two separate fixes.

**Permission prompts** — root cause is that `.claude/settings.local.json` allowlists only the 9 MCP tools codelens uses. Every Bash command (`rg`, `git --no-pager diff`, `ast-grep`, `biome`, `tsc`, `fallow`, `wc`, `timeout`) is a prompt. Fix: add a **scoped Bash allowlist** (see §4.1). Scope is critical — never allowlist `Bash(*)`, only the exact commands and flag shapes codelens issues.

**Output-format drift** — schemas already exist (`schema/reviews-entry.schema.json`, `schema/INTERNAL.md`) with `additionalProperties: false` and 13 required fields. Drift happens because there is **no validation gate** between "agent wrote the entry" and "entry appended to reviews.json." Fix: add a deterministic `ajv` (or `ctx_execute`-inline) validation step in Phase 4 (see §4.2). Same applies to the markdown report — add a structural lint (title regex + severity-header regex + required-section-order check).

## 3. Design Principles (v0.0.8)

Carried forward from v0.0.6 (validated, no change):
- **Single-pass per hotspot** — never re-read.
- **Domain-aware** — only run requested domains.
- **Scope-aware** — every command targets `scopePath`.
- **Severity-first ordering** — Critical > High > Medium > Low > Informational.
- **Evidence-backed** — every finding has `file:line` + snippet.
- **Cross-domain dedup** — same `file:line` (±2) merges.
- **Exclusions honored** — `.claude/codelens-exclusions.json`.
- **Append-only log** — one entry per review.

New for v0.0.8:
- **Deterministic over instructive.** Where a behavior can be enforced by tooling (schema validation, allowlist, terminal guard), it MUST be. Constraints remaining in the prompt should be only those with no tooling equivalent (e.g., "discard low-confidence findings" — judgment, not mechanical).
- **Phase 3 correlates, does not re-discover.** Phase 2's signals are the input to Phase 3. Phase 3's job is to verify and localize, not to re-run searches that Phase 2 already ran.
- **No prompt should grow to compensate for a missing gate.** If a constraint has been modified in 3+ consecutive versions, the constraint is in the wrong layer.

## 4. Proposed Changes

### 4.1 Permission hygiene — scoped Bash allowlist

**Problem:** `.claude/settings.local.json` currently allowlists 9 MCP tools and zero Bash commands. Every `rg`, `git`, `ast-grep`, `biome`, `tsc`, `fallow`, `wc`, `timeout` invocation prompts.

**Change:** Add a scoped Bash allowlist to `.claude/settings.local.json`. Patterns must be tight enough to avoid prompt-fatigue without opening arbitrary command execution.

```jsonc
{
  "permissions": {
    "allow": [
      // existing MCP allows (unchanged)
      "mcp__plugin_context-mode_context-mode__ctx_batch_execute",
      "mcp__plugin_context-mode_context-mode__ctx_execute",
      "mcp__plugin_context-mode_context-mode__ctx_execute_file",
      "mcp__plugin_context-mode_context-mode__ctx_search",
      "mcp__plugin_context-mode_context-mode__ctx_stats",
      "mcp__plugin_context-mode_context-mode__ctx_fetch_and_index",
      "mcp__plugin_context-mode_context-mode__ctx_index",
      "mcp__plugin_context-mode_context-mode__ctx_doctor",
      "mcp__plugin_context-mode_context-mode__ctx_purge",
      "mcp__plugin_context-mode_context-mode__ctx_insight",
      "mcp__plugin_context-mode_context-mode__ctx_upgrade",
      "mcp__plugin_context7_context7__resolve-library-id",
      "mcp__plugin_context7_context7__query-docs",
      "WebSearch",

      // NEW: scoped Bash allows for codelens review/doctor commands
      "Bash(rg:*)",
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git rev-parse:*)",
      "Bash(git show:*)",
      "Bash(ast-grep:*)",
      "Bash(sg:*)",
      "Bash(biome:*)",
      "Bash(npx --no-install tsc:*)",
      "Bash(npx --no-install biome:*)",
      "Bash(fallow:*)",
      "Bash(wc:*)",
      "Bash(timeout:*)",
      "Bash(cat schema/*)",
      "Bash(cat .claude/codelens-exclusions.json)",
      "Bash(cat .claude/review-presets.json)"
    ]
  }
}
```

**Safety screen:**
- No `Bash(*)` — never blanket-allow.
- No `rm`, `mv`, `>`, `>>`, `curl`, `wget`, `npm install` (mutating/network operations stay prompt-gated).
- `git` is limited to read-only subcommands (`status`, `diff`, `log`, `rev-parse`, `show`) — no `commit`, `push`, `reset`, `checkout`.
- `npx --no-install` only — no implicit installs.
- The `fewer-permission-prompts` skill in this repo can scan transcripts and propose additional patterns if prompts persist after this change.

**Cost:** One-time additions to `.claude/settings.local.json`. No agent change. **Risk:** low — patterns are scoped.

**Verification:** After the change, run `/codelens:review` on a fixture and confirm zero Bash permission prompts appear. The `fewer-permission-prompts` skill output should show <5 unprompted Bash calls remaining.

### 4.2 Output drift — deterministic validation gate

**Problem:** The agent is instructed to emit reviews.json entries conforming to `schema/reviews-entry.schema.json`. There is no check that it did. Schema drift is silent.

**Change:** Add a validation step at the end of Phase 4, before the append. Two checks:

**A. JSON Schema validation (reviews.json entry).** Use `ajv` via `ctx_execute` (keeps the validator out of the agent's context — only pass/fail returns):

```javascript
// ctx_execute, language: javascript
// Reads the candidate entry from a tmp file, validates against the schema, prints "OK" or the error list.
const fs = require('fs');
const Ajv = require('ajv'); // or require from a vendored path / npx --no-install
const entry = JSON.parse(fs.readFileSync(process.env.ENTRY_PATH, 'utf8'));
const schema = JSON.parse(fs.readFileSync('schema/reviews-entry.schema.json', 'utf8'));
const ajv = new Ajv({allErrors: true});
const valid = ajv.validate(schema, entry);
if (!valid) {
  console.log('FAIL: ' + ajv.errorsText());
  process.exit(1);
}
console.log('OK');
```

**Concern:** `ajv` is a runtime dependency the project does not currently have (codelens is "Markdown only, no build step"). **Mitigation:** run the validator inside `ctx_execute`'s sandbox (which already has Node + npm resolution), or fall back to a hand-written structural check (13 required keys present + `command` matches `^/review` + `status` in enum + `reviewerVersion` matches semver). The hand-written check is 30 lines of JS, no deps, and covers the high-frequency drift modes. Recommend the hand-written check for v0.0.8; revisit `ajv` if drift persists.

**B. Markdown report structural lint.** The report must have: title `# Codebase Analysis Report: ...`, severity sections in order, required non-severity sections (What's Done Well, Priority Actions, Methodology). Run as a `ctx_execute` shell post-check:

```sh
# ctx_execute, language: shell
# Prints OK or the first missing/unordered element.
report="$REPORT_PATH"
grep -q '^# Codebase Analysis Report:' "$report" || { echo 'FAIL: title'; exit 1; }
# section-order check: extract H2 order, assert severity comes before What's Done Well
order=$(grep -E '^## ' "$report" | head -20)
echo "$order" | grep -q 'What.s Done Well' || { echo 'FAIL: missing What's Done Well'; exit 1; }
echo 'OK'
```

**Gate behavior:** If either check fails, the agent MUST NOT append to reviews.json. It prints a `STATUS: partial` line and lists the validation error. This converts silent drift into a loud, recoverable failure.

**Constraint reduction:** Once the gate exists, several prompt constraints become redundant and can be removed from the agent body:
- "The appended entry MUST validate against `schema/reviews-entry.schema.json`" — replaced by the gate.
- "If any field can't be populated ... set its value to `0` / `unknown`" — replaced by the gate's error message, which tells the agent exactly what's missing.
- "Never invent data" — still a judgment constraint, keep.

**Cost:** ~30 LOC of validation script (in `scripts/validate-review.{js,sh}` or inline). Agent body shrinks by ~5 constraint lines.

### 4.3 Phase 3 — correlator, not re-discoverer

**Problem (diagnosed in architecture review):** Phase 2 runs `p2-sec-patterns`, `p2-sec-secrets`, `p2-quality`, `p2-a11y-*` searches. Phase 3 then re-runs ast-grep/rg patterns on the same hotspots for the same signals. The architecture review's diagram captures this:

```
current:  Tools Discover → Phase 3 Rediscovers → Deduplication → Report
target:   Tools Discover → LLM Correlates → LLM Prioritizes → Report
```

**Change:** Phase 3's input is the Phase 2 output index, not a fresh search. Phase 3's commands become **file-targeted verifications**, not scope-wide re-searches:

- For each hotspot file, run ast-grep on **that file only** (`ast-grep -p <pattern> <file>`), not the whole scope.
- The signal that triggered the hotspot (from Phase 2's `density[path]`, `complexity[path]`, `centrality[path]`) is already known — Phase 3 confirms and localizes the line numbers.
- Drop Phase 3 commands that duplicate Phase 2 signals (e.g., if Phase 2's `p2-sec-patterns` already caught `localStorage.setItem`, Phase 3 should not re-grep for it — it should ast-grep the specific file to get the precise line).

**Coverage matrix update:** The existing coverage matrix (old regex → new source) stays, but each row is now annotated with **"discovered in Phase 2, localized in Phase 3"** vs **"discovered and localized in Phase 3"** (the latter only for patterns ast-grep can find but rg-in-Phase-2 cannot — currently a small set).

**Token impact:** Phase 3 per-hotspot commands shrink (file-targeted, not scope-targeted). The architecture review's "Phase 3 contains too much analyzer logic inside prompt text" risk is directly addressed.

**Cost:** Rewrite of Phase 3 command-construction block. ~40 LOC change in the agent. No new dependencies.

**Risk:** If Phase 2 misses a signal (e.g., a pattern ast-grep catches but rg does not), Phase 3 will not catch it either under the new model. **Mitigation:** Phase 2's pattern list must be the superset. Audit Phase 3's current ast-grep patterns and ensure each has a Phase 2 rg equivalent (or is explicitly marked "Phase 3 only — no Phase 2 equivalent"). The v0.0.6 coverage matrix already does most of this work.

### 4.4 Terminal guard — prevent re-execution after output

**Problem (research note):** "After generating output files, the agent may execute again and rewrite files."

**Root cause:** No explicit terminal signal. The agent finishes Phase 4, but nothing prevents it from re-entering Phase 0 in the same turn.

**Change:** Add a `Stop` hook in `.claude/settings.local.json` (or the agent's own frontmatter) that checks whether a report was written this turn and, if so, blocks further tool calls with a "review complete — start a new turn" message. Alternatively (simpler, lower-risk), add a final instruction at the end of Phase 4:

```
END OF WORKFLOW. Do not re-enter Phase 0. Do not re-run any command.
If the user wants another review, they will issue a new /codelens:review call.
```

This is a prompt-level fix, which contradicts the "deterministic over instructive" principle — but a true Stop hook for "agent wrote a file, now halt" is fiddly to scope correctly (the agent writes multiple files per review). **Recommendation:** ship the prompt-level instruction for v0.0.8, add the Stop hook in v0.0.9 if the instruction proves insufficient.

**Cost:** 3 lines at the end of the agent body. No new tooling.

---

### 4.5 Constraint diet — remove prompt rules replaced by gates

Once §4.2 (validation gate) and §4.1 (allowlist) land, the following constraints in the agent body become redundant and should be removed:

| Current constraint | Why redundant |
|---|---|
| "The appended entry MUST validate against `schema/reviews-entry.schema.json`" | Gate enforces. |
| "If any field can't be populated ... set to `0` / `unknown`" | Gate error tells the agent. |
| "Always use ctx_batch_execute for Phase 1+2" | The single-batch pattern is now the only pattern — no need to instruct it. |
| "Never use Glob when rg can do the job" | Glob is not in the agent's tool list anymore. The constraint references a non-option. |
| "Always use native Write tool for final report" | Only Write can write — ctx_execute sandbox doesn't persist. Constraint is a tautology. |

**Keep** constraints that have no tooling equivalent: "Discard low-confidence findings" (judgment), "Keep the report actionable" (judgment), "Never analyze domains not in config.domains" (the gate cannot know which domains were requested without scope plumbing — defer to v0.0.9).

**Target:** Reduce behavioral constraints from ~20 to ~10. Each removed constraint is a row in the v0.0.8 changelog.

## 5. Migration Path (v0.0.7 → v0.0.8)

**Incremental, low-risk ordering.** Each step ships independently; any step can be reverted without unwinding the others.

1. **v0.0.8-alpha — Permission allowlist (§4.1).** Smallest change, highest UX impact. Ship first to clear the prompt noise for all subsequent testing.
2. **v0.0.8-beta — Validation gate (§4.2).** Add `scripts/validate-review.sh` (markdown lint) and the inline JS structural check. Gate runs after Phase 4 write, before append. No agent-body change yet — just the gate exists.
3. **v0.0.8-rc.1 — Wire gate into agent (§4.2 + §4.5).** Edit Phase 4 to call the gate. Remove the ~5 redundant constraints identified in §4.5.
4. **v0.0.8-rc.2 — Phase 3 correlator shift (§4.3).** Rewrite Phase 3 command construction. Smoke-test on 3 codebases (codelens itself, a small Next.js app, a medium React+TS codebase) — this is the v0.0.7 Priority A validation debt, finally paid.
5. **v0.0.8 — Terminal guard + release (§4.4).** Add the end-of-workflow instruction. Bump version, write CHANGELOG.

**Estimated effort:** 1–2 days for steps 1–3 (mechanical), 2–3 days for step 4 (requires smoke-test validation), 0.5 days for step 5. Total: ~1 week of focused work.

**Smoke test gate (must pass before release):**
- `/codelens:doctor` — 13 checks still pass.
- `/codelens:review` on codelens itself — report generates, reviews.json entry validates, zero Bash prompts with the new allowlist.
- `/codelens:review security src/auth` on a fixture with known security issues — findings match expectations.
- `/codelens:review` on a non-JS/TS codebase (e.g., a Python repo) — Language Support Note appears, no biome/tsc errors leak.

## 7. Risks and Tradeoffs

| Risk | Severity | Mitigation |
|---|---|---|
| **R1: Scoped Bash allowlist too narrow** — agent falls back to a command shape not on the list, prompt returns. | Low | Iterate using `fewer-permission-prompts` skill output after first smoke test. Adding patterns is safe. |
| **R2: Hand-written schema check misses a drift mode** — agent emits a structurally-valid-but-semantically-wrong entry. | Medium | Gate is a floor, not a ceiling. The judgment constraints ("Discard low-confidence findings") remain. Audit drift modes after 1 month of real reviews, promote to gate as needed. |
| **R3: Phase 3 correlator shift drops a signal** — a pattern Phase 2's rg misses but Phase 3's ast-grep catches. | Medium | Audit coverage matrix (§4.3). Any "Phase 3 only" pattern must be explicitly marked and justified. Smoke test on fixture with known patterns. |
| **R4: Terminal-guard instruction ignored** — agent re-enters Phase 0 anyway. | Low | Escalate to Stop hook in v0.0.9 if observed. |
| **R5: Constraint diet removes a constraint still doing work** — a constraint that *appeared* redundant was actually catching a case the gate doesn't. | Medium | Each removed constraint gets a changelog row naming what replaces it. Re-add on first regression. |
| **R6: Scope creep into subagent split** — the research note's "monolithic" framing pulls toward a rewrite the evidence doesn't justify. | High | This doc exists specifically to anchor the decision. Re-read §2 Q1–Q2 before any structural change. |

## 10. Open Questions

1. **Validator home.** Should `scripts/validate-review-entry.js` be a checked-in file, or inlined in the agent body as a `ctx_execute` code block? Checked-in is more maintainable; inlined avoids a new file. **Recommendation:** checked-in, called via `ctx_execute_file` — keeps the agent body lean.
2. **Allowlist scope for `git`.** Current proposal allows `git status/diff/log/rev-parse/show`. Should `git blame` be added? It's read-only but not currently used by the agent. **Recommendation:** no — add only when the agent uses it.
3. **Phase 3 parallelization (Q3).** Parallelizing per-hotspot batches into one `ctx_batch_execute` is a v0.0.8 candidate but is an execution-model change layered on top of §4.3. **Recommendation:** ship §4.3 first, measure wall-clock, parallelize in v0.0.9 if warranted.

## Appendix B — What this design does NOT do (and why)

- **Does not split the agent into orchestrator + subagents.** Evidence (Q2, Q4) shows the context cost of replicating state into subagents exceeds the parallelism win at current scale. The subagent split is a v0.1.0 candidate, gated on a measured Phase 3 bottleneck.
- **Does not add a build step or runtime dependency.** Codelens is "Markdown only." The validators are plain JS/shell scripts called via `ctx_execute` sandbox — they do not introduce a build step or a runtime dep the user must install.
- **Does not change the schema.** `schema/reviews-entry.schema.json` and `schema/INTERNAL.md` are v0.0.7-stable. v0.0.8 enforces them; it does not modify them.
- **Does not add new domains, languages, or analyzers.** v0.0.8 is a discipline release. Feature additions resume in v0.0.9.
- **Does not rewrite the skill dispatchers.** `/codelens:review` and `/codelens:doctor` are already thin (38 LOC, 47 LOC). No change.
