# codelens v0.0.10 — Extensibility + Self-Contained Release

**Date**: 2026-06-19
**Status**: Design — approved
**Target version**: 0.0.10 (beta — no backward compatibility)
**Prior specs**: [v0.0.8 architecture](./2026-06-18-codelens-v0.0.8-architecture-design.md), [v0.0.9 gate hardening](./2026-06-19-codelens-v0.0.9-gate-hardening-design.md)
**Supersedes**: the earlier v0.0.10 "doc-cleanup-only" draft (scope expanded after user feedback)

---

## 1. Problem

The plugin is at v0.0.9 beta. Four structural problems block it from being manageable or production-ready:

1. **Documentation drift** — every primary doc states facts that contradict the shipped code (6-field `reviews.json` vs actual 11-field `reviews.log`; v0.0.7 CLAUDE vs v0.0.9 manifest; "no phase gates" vs Phase 4's mandatory `STATUS:` gates).
2. **Silent external dependencies** — `plugin.json` declares no `mcpServers` and no `permissions`. Users must manually install 2 MCP servers + 5 CLIs (rg, biome, fallow, tsc, ast-grep). Fresh installs fail confusingly at Phase 0.
3. **Hardcoded architecture** — all check logic lives inline in the 450-line agent. Adding a company-specific check (e.g., ".env.example must exist") requires editing the agent in 3 places. Adding a language (Python, PHP) requires rewriting Phase 1+2 commands and Phase 3 ast-grep flags. No config-driven extensibility exists.
4. **Stack-blind doctor** — `/codelens:doctor` checks biome/tsc/fallow/ast-grep unconditionally even on a Python repo, producing irrelevant warnings.

Plus smaller issues: a diff-scope `scopePath` word-splitting bug (P0), a template that violates its own abstraction rule, token redundancy in the agent, and no schema versioning on `reviews.log`.

## 2. Goals

1. **Eliminate doc drift** — every doc reflects v0.0.10 reality.
2. **Self-contained install** — `/plugin install codelens` auto-provisions MCP servers; npm CLIs auto-fetch via `npx` on first use. Only `rg` remains user-installed (native binary, can't bundle).
3. **Config-driven extensibility** — `config/custom-checks.json` for company-specific evidence-based checks; `config/languages.json` for multi-language support (mechanism now, JS/TS populated, Python/PHP via future config edits).
4. **Stack-aware doctor** — detects project stack, skips irrelevant tool checks.
5. **Drop backward compatibility** — beta, no official release. `reviews.log` is the canonical shape. `schema` field is required. No migration prose.
6. **Beta framing everywhere** — README and user-facing docs state "v0.0.10 beta — no backward compatibility guaranteed; shape may change before v1.0."
7. **Token efficiency** — agent prompt ≥25% smaller via shared severity ladder + constraints dedup.

## 3. Non-goals (deferred to v0.1.0+)

- **Populating Python/PHP in `config/languages.json`** — the mechanism ships in v0.0.10; the actual Python lint/typecheck commands and PHP/Laravel patterns are follow-up PRs (config edits only, no agent changes).
- **Judgment-based custom checks** — `config/custom-checks.json` supports evidence-based checks only (shell command → finding). Judgment-based rules ("this abstraction is wrong") stay in human review; they don't scale as injected prose.
- **Hooks** — plugin-dev conventions say on-demand plugins (skill+agent dispatch) don't need session-lifecycle hooks. No hooks added. (Investigated PreToolUse, PostToolUse, Stop, SessionStart — none add value for codelens without violating scope/latency guidance.)
- **Bundling native binaries** — `rg` can't be bundled (per-OS/arch matrix, no plugin postinstall hook). Stays user-installed.
- **Runtime token-cost benchmarking** — current `scripts/bench-phase.sh` measures static prompt bytes (`chars/4`), not runtime tokens. A real token-API bench is deferred.
- **Gemini's orchestrator+worker decomposition** — rejected (contradicts the v0.0.5→v0.0.9 consolidation arc).
- **Auto-write settings.json** (Gemini's TASK 2) — rejected (unsafe side effect for a diagnostic command).

### Part B — Diff-scope `scopePath` correctness (P0)

`agents/codelens-reviewer.md` Phase 1+2 maps `scopePath = git diff --name-only` for diff scope — multi-line output that breaks every command substituting `<scopePath>` as one path arg.

### Part C — Manifest + bundled tools

#### C1. Inline MCP servers into `plugin.json`

Add `mcpServers` so `/plugin install codelens` auto-provisions everything:
```json
"mcpServers": {
  "context-mode": { "type": "stdio", "command": "npx", "args": ["-y", "context-mode"] },
  "context7":     { "type": "stdio", "command": "npx", "args": ["-y", "@upstash/context7-mcp"] }
}
```
**Verify before commit**: `npx -y @upstash/context7-mcp --help` exposes binary `context7`; confirm tool-name registration form (`mcp__plugin_<server>_<server>__*` vs `mcp__<server>__*`).

#### C2. Permissions allowlist

Valid `Bash(<first-token>:*)` grammar only — Claude Code matches only the first whitespace-delimited token. Collapse the invalid `Bash(npx --yes ... tsc:*)` and `Bash(./node_modules/.bin/tsc:*)` to `Bash(npx:*)`. Fix the earlier `context_mode` → `context-mode` typo. Remove unnecessary `Bash(bash scripts/...)` (validator runs via `ctx_execute`, not Bash).

#### C3. Wrap npm CLIs in `npx` with binary fast-path

The agent already uses `binary || npx` for tsc (line 165). Replicate for the other 3 npm CLIs so users don't pre-install them:
- **biome**: `command -v biome >/dev/null 2>&1 && biome lint ... || npx --yes @biomejs/biome lint ...`
- **fallow**: `command -v fallow >/dev/null 2>&1 && fallow ... || npx --yes fallow ...` (verify package name)
- **ast-grep**: `command -v sg >/dev/null 2>&1 && sg ... || npx --yes @ast-grep/cli run ...`

**rg stays user-installed** — native binary, per-OS/arch matrix, no plugin postinstall mechanism.

Doctor checks for these tools change from "install required" → "auto-fetched via npx on first use; faster if pre-installed" (still WARN-level; first-run npx fetch adds 5–30s).

### Part G — Stack-aware doctor

Doctor detects the project's primary stack and gates optional-tool checks on relevance.

1. **Detection** (new early check, after rg/git): probe manifest files → `detectedStack` ∈ `{js-ts, python, php, go, rust, unknown}`. Reads `config/languages.json` (Part I) for the manifest→stack mapping.
2. **Gate optional checks**: biome/tsc/fallow/ast-grep only run when `detectedStack === 'js-ts'`. On other stacks, print `[SKIP] <tool> not applicable for <stack> projects` (not WARN). Critical checks (rg, git, context-mode, config files) always run.
3. **Future**: when Python support is populated in `languages.json`, ruff/mypy checks fire only on `python` stack.

### Part H — `config/custom-checks.json` (evidence-based custom checks)

New config file + agent integration. Schema:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "checks": [
    {
      "id": "env-example-exists",
      "domain": "security",
      "severity": "High",
      "title": ".env.example file must exist in project root",
      "detect": "test -f .env.example || echo 'MISSING .env.example'",
      "passSignal": "OK",
      "description": "Ensures developers have a template for required environment variables."
    }
  ]
}
```

**Field rules**:
- `id` — kebab-case, unique. Becomes the finding's rule identifier in the report.
- `domain` — `security | architecture | quality | a11y`. Must match an active domain or the check is skipped.
- `severity` — `Critical | High | Medium | Low | Informational`.
- `detect` — shell command. Output is the evidence.
- `passSignal` — string that, if present in output, means "passed." Default `"OK"`. If `detect` outputs nothing and no `passSignal` set, treated as pass.
- `title`, `description` — human-readable, appear in the report.

**Agent integration** (3 edits):
1. **Phase 0.5** (new, after preflight): `ctx_execute` reads `config/custom-checks.json` via `fs.readFileSync`, prints check count. Missing/empty file → skip silently. Pattern mirrors the G1 gate-load (agent lines 316–343).
2. **Phase 1+2 batch**: for each custom check whose `domain` ∈ `config.domains`, append `{label: "custom-<id>", command: "<detect>"}` to the `ctx_batch_execute` commands array.
3. **Phase 4 report build**: for each custom check, if output doesn't match `passSignal`, emit a finding row in the severity section using `id`/`title`/`description`.

**Doctor**: new check 18 (critical) — `config/custom-checks.json` valid JSON if present. File is optional (users without custom checks have none).

**Validator**: new script `scripts/validate-custom-checks.js` — validates schema (id kebab-case + unique, domain/severity/detect non-empty, severity ∈ allowed set). Run from doctor.

**Ship 2 example checks**: `env-example-exists` + `readme-exists` to demonstrate the pattern.

### Part I — `config/languages.json` (multi-language mechanism)

**Scope**: build the infrastructure + generalized command-building now. Ship JS/TS fully populated. Python/PHP are placeholder entries — populated in follow-up PRs via config edits only (no agent changes).

Schema (abbreviated — full version in plan):
```json
{
  "languages": {
    "js-ts": {
      "extensions": ["js","jsx","ts","tsx"],
      "manifestFiles": ["package.json"],
      "lint": { "command": "biome lint <scope> --reporter=json", "npxFallback": "npx --yes @biomejs/biome lint ...", "binaryCheck": "command -v biome >/dev/null 2>&1", "notAvailableSignal": "biome-not-available" },
      "typecheck": { "command": "tsc -p . --noEmit --skipLibCheck --pretty false", "npxFallback": "npx --yes --package=typescript tsc ...", "binaryCheck": "test -x ./node_modules/.bin/tsc", "notAvailableSignal": "tsc-not-available" },
      "deadCode": { "command": "fallow dead-code --format=json", "npxFallback": "npx --yes fallow ...", "binaryCheck": "command -v fallow >/dev/null 2>&1", "notAvailableSignal": "fallow-not-available" },
      "astGrepLang": "tsx,jsx,ts,js",
      "severityMappings": { "lint/a11y/*": "a11y-High", "TS2xxx": "quality-High", "...": "..." },
      "phase3Patterns": { "xss-innerhtml": "$$.innerHTML", "...": "..." }
    },
    "python": { "_status": "placeholder — populate in follow-up PR", "extensions": ["py"], "manifestFiles": ["requirements.txt","pyproject.toml"], "lint": {"_todo": "ruff"}, "typecheck": {"_todo": "mypy"} },
    "php": { "_status": "placeholder — Laravel support", "extensions": ["php"], "manifestFiles": ["composer.json"], "lint": {"_todo": "php -l or artisan"}, "typecheck": {"_todo": "psalm or phpstan"} }
  }
}
```

**Agent integration** (replaces hardcoded JS/TS commands):
1. **Phase 0.5**: also load `config/languages.json`.
2. **Stack detection** (Phase 1, replaces binary logic at line 199): `primaryLang = first language whose manifestFiles exist`. `unknown` if none match.
3. **Phase 1+2 batch**: build `r3-complexity`/`p2-biome`/`p2-tsc`/`p2-fallow-*` from `primaryLang`'s config — `<lint.binaryCheck> && <lint.command> || <lint.npxFallback>`. No hardcoded tool strings.
4. **Phase 3**: ast-grep `-l` from `primaryLang.astGrepLang`; patterns from `primaryLang.phase3Patterns`. `unknown` → rg-only fallback.
5. **Severity mapping** (Phase 4): `primaryLang.severityMappings` replaces the hardcoded table at line 215.

**Doctor** reads `languages.json` to know which tools are relevant for the detected stack (Part G).

## 8. Risks & mitigations

- **Permissions grammar wrong** → plugin fails to load or silently no-ops. **Mitigate**: validate with `claude plugin validate`; deliberately typo one entry to confirm allowlist is read.
- **MCP tool-name registration form differs** (`mcp__<server>__*` vs `mcp__plugin_<server>_<server>__*`) → every MCP entry in C2 AND agent's `tools:` array breaks. **Mitigate**: test on clean install after C1 lands; update en masse if needed.
- **`ctx_execute` temp-file persistence** — verified working for `/tmp` writes. **Mitigate**: Part B's mechanism is viable; PID suffix handles concurrency.
- **Token-efficiency refactor shifts severity assignments** — **Mitigate**: smoke-test security + a11y before/after; gate is ±1 per category, no Critical↔Low drift.
- **Custom-checks shell injection** — `detect` commands run as Bash; a malicious config could execute anything. **Mitigate**: file is user-authored in their own repo (same trust level as their own scripts); document the trust assumption in `templates/README.md` and `CONTRIBUTING.md`. Doctor's check 18 validates JSON structure, not command safety.
- **Languages.json breaking JS/TS regression** — refactoring hardcoded commands to config-driven could introduce subtle differences. **Mitigate**: verification step 7 (severity regression check); JS/TS `languages.json` entry is a faithful transcription of current behavior.
- **Beta framing scares users** — "no backward compat" may reduce adoption. **Mitigate**: this is honest; the alternative (implying stability we can't guarantee) is worse. v1.0 is the stability commitment.
