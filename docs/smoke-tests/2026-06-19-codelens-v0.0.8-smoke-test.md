## Issues Found (sorted by severity)

### 🔴 Issue #1 — CRITICAL: Agent ignored the new 6-field reviews.json schema

**What happened:** The agent wrote a 6-key entry to `my-portfolio/.codelens/reviews.json`, but the keys are wrong:

| Agent wrote | Schema requires |
|---|---|
| `timestamp` ✅ | `timestamp` |
| `scope` ✅ | `scope` |
| `scopeTarget` ❌ | (not in schema) |
| `domains` ❌ | (not in schema) |
| `outputFile` ❌ | `reportPath` |
| `summary: {critical, high, medium, low, informational, total}` ❌ | `findings: {critical, high, medium, low, informational}` |
| `topFindings: [string...]` ❌ | (not in schema) |
| `notes` ❌ | (not in schema) |
| (missing) ❌ | `summary` (one-line string) |
| (missing) ❌ | `reviewerVersion` |

`validate-entry.js` against the agent's output: **FAIL: missing required field findings**.

**Root cause:** The agent didn't load `templates/reviews-entry.json` to learn the shape. It used its own internal model of what an entry should look like — which is stale (matches the v1.x heavy schema, not the v0.0.8 minimal one).

**Fix:** Phase 4 instructs `ctx_execute_file path: "templates/reviews-entry.json"` but the agent didn't run that call (transcript shows 0 references to `templates/`). The instruction exists in the agent body but isn't being followed.

### 🔴 Issue #2 — CRITICAL: Agent didn't load `templates/report.md`

**What happened:** `grep 'templates/' transcript` returned **0 hits**. The agent didn't load the report template at any point during the run.

**Evidence in the output:** The agent produced its own report format that doesn't match `templates/report.md`:

| Templates/report.md says | Agent produced |
|---|---|
| `## Scorecard` with `Severity | Count | Domain | Count` columns | `## Scorecard` with `Domain | Score | Notes` columns (letter grades B/C+) |
| `## What's Done Well` per requested domain | ✅ Got this right |
| `## Priority Actions` with Immediate/Short-Term/Medium-Term/Backlog | (check report — likely missed) |
| `## Methodology` with prose + per-domain table | ✅ Got the prose; missed the table |
| Title format: `**Date:** ... **Stack:** ... **Scope:** ... **Reviewer:**` | Different: `**Project**: ... **Stack:** ... **Scope:** ... **Domains:** ... **Reviewed:**` |

The agent reinvented the format from training data + the prior-version reports it has seen, instead of consulting the template.

**Root cause:** Same as Issue #1 — Phase 4 instructions tell the agent to load `templates/report.md` via `ctx_execute_file`, but it didn't. The instructions exist; the agent isn't following them.

### 🔴 Issue #3 — CRITICAL: Validation gates did NOT fire

**What happened:** `grep -c 'validate-entry|validate-report|STATUS:' transcript` returned **0**. The agent didn't run either validator script during Phase 4.

**Consequence:** Had the gates fired, they would have caught Issue #1 (entry validator would FAIL on the wrong shape) and Issue #2 (report validator might catch the scorecard column mismatch if we tighten the check). The gates are wired into Phase 4 prose but the agent skipped them.

**Root cause:** Phase 4 has a "Validation gates" subsection instructing the agent to run validators via `ctx_execute`. The agent made 0 `ctx_execute` calls during the entire run (only `ctx_batch_execute` and `ctx_execute_file`). It treated the gate instructions as descriptive prose, not required steps.

**Implication:** The deterministic-enforcement round shipped gates that aren't being invoked. Pain #2 (output drift) is **still unaddressed in practice** despite the validators existing on disk.

### 🟡 Issue #4 — MEDIUM: Agent wrote `topFindings` array — useful but not in schema

**What happened:** The agent's entry includes `topFindings: [string...]` with 8 one-line summaries of the highest-severity findings. This is genuinely useful for trend scanning.

**Implication:** The minimal 6-field schema may be too minimal. Real users (and the agent itself) want the top findings visible without opening the full report.

**Decision needed:** add `topFindings` (optional, max 10 items) to the schema, or keep the schema strict and lose this signal. Recommend adding it — it's the only lossy-compression field that pays for itself.

## Summary: Agent Improvements Needed

Ranked by impact:

1. **Make Phase 4 actually load templates.** The instructions say `ctx_execute_file path: "templates/report.md"` and `ctx_execute_file path: "templates/reviews-entry.json"` — the agent ignored both. Either:
   - **(a) Harden the instructions** — change "consult before writing" to "your FIRST action in Phase 4 MUST be `ctx_execute_file` on these paths; do not proceed until both return."
   - **(b) Inline-cite the schema** — embed the 6-field shape directly in the agent body so the agent can't drift even if it skips the load. (Loses the benefit of externalizing, but works.)
   - **(c) Add a gate that blocks the append if templates weren't loaded** — track whether `ctx_execute_file` was called on those paths; if not, refuse to append.

   Recommend (a) + (c) together: stronger instruction + a runtime check.

2. **Make validation gates actually fire.** Same problem as #1 — the instructions exist but aren't followed. Same fixes apply: harden the language ("MUST run validators before append; on FAIL do not append") + verify the agent ran them.

3. **Decide on `topFindings` field.** The agent wants to write it. Either add it to the schema (recommended — useful for trend scanning) or strengthen the "additionalProperties: false" enforcement to make the agent stop.

4. **Scorecard format needs enforcement.** The report validator currently checks for `## Scorecard` header presence but not the column shape. The agent produced `Domain | Score | Notes` instead of `Severity | Count | Domain | Count`. Tighten `validate-report.sh` to check for the severity-count table specifically.

5. **(Not from this smoke test but worth noting)** The agent's terminal guard instruction we added in round 4 — verify it actually prevents re-entry. This smoke test was a single-turn headless run, so re-entry wasn't tested.
