## What changed vs v0.0.8 (the win)

v0.0.8 failed because the agent **treated gate instructions as descriptive prose** — 0 `ctx_execute` calls, 0 `templates/` references, 0 validator invocations. The agent simply didn't try.

v0.0.9 fails for a **completely different and fixable reason**: the agent **did try** to run the gates, but the gate calls themselves have implementation bugs. Concretely, the transcript shows the agent issued:

- ✅ All 3 Step 1 `ctx_execute_file` calls, **verbatim** matching the JSON blocks in the spec (Calls #2, #3, #4 — `"path":"templates/report.md"`, `"path":"templates/reviews-entry.json"`, `"path":"templates/README.md"` with the exact `LOADED ...` code strings)
- ✅ 3 `ctx_execute` calls (Calls #1, #5, #6)
- ✅ 9 references to `CLAUDE_PROJECT_DIR` — the agent learned to use the env var
- ✅ 42 references to `templates/` paths (was 0 in v0.0.8)
- ✅ The agent emitted `STATUS: partial` 3 times — it understood the marker convention and self-reported failure

**The preflight banner + real JSON tool-call blocks worked.** The agent is no longer ignoring the gates; it's trying to run them. That is the core Phase A hypothesis, and it is validated.

## The two bugs that prevent the gates from firing

### 🔴 Bug 1 — Step 1 paths resolve against the target repo, not the plugin

**What happened:** All 3 Step 1 `ctx_execute_file` calls errored (exit code 1). The transcript shows the actual resolution:

```
const FILE_CONTENT_PATH = "/home/nurmdrafi/Desktop/MyProject/my-portfolio/templates/report.md";
```

`ctx_execute_file` resolved `path: "templates/report.md"` against **my-portfolio** (the target repo being reviewed, which is the headless run's cwd), not the **codelens** plugin repo (where `templates/` actually lives). my-portfolio has no `templates/` dir, so `readFileSync` threw.

**Why my spec missed this:** I verified Steps 4 and 6 use `CLAUDE_PROJECT_DIR` to locate the validator scripts (and confirmed in the sandbox that the env var is set). But for Step 1 I left the `path` field as the bare repo-relative string `"templates/report.md"` — assuming `ctx_execute_file` would resolve it from the plugin root. It doesn't. It resolves from the runtime cwd, which for a code-review plugin is the *target repo*.

This is the same root-cause class as the `require('./scripts/...')` failure I caught earlier in sandbox testing — paths inside the plugin must be rooted via `CLAUDE_PROJECT_DIR`, not left relative.

**Fix:** Step 1's three `ctx_execute_file` calls must use absolute paths via `CLAUDE_PROJECT_DIR`. Either:
- (a) Inline the absolute path in the `path` field: `"$CLAUDE_PROJECT_DIR/templates/report.md"` (needs verification whether `ctx_execute_file`'s `path` field expands env vars), or
- (b) Switch Step 1 from `ctx_execute_file` to `ctx_execute` with `language: "javascript"` and use `fs.readFileSync(process.env.CLAUDE_PROJECT_DIR + '/templates/report.md', 'utf8')` directly — the same pattern Steps 4/6 use.

Option (b) is safer because we've already verified that pattern works in the sandbox. Recommend (b).

### 🔴 Bug 2 — Agent improvised its own logic when Step 1 errored

**What happened:** Call #5 and Call #6 in the transcript show the agent wrote its **own** ad-hoc JavaScript to (a) inspect the filesystem and (b) append to `reviews.log` — instead of issuing the exact `bash "$CLAUDE_PROJECT_DIR/scripts/validate-report.sh" ...` and `require(process.env.CLAUDE_PROJECT_DIR + '/scripts/validate-entry.js')` blocks specified in Steps 4 and 6.

The agent saw Step 1 fail, concluded "templates/scripts not installed in environment" (its own words in the `methodology` field of the entry), and decided to handle Phase 4 itself. It bypassed the gates entirely — including Step 7's precondition check (which would have refused the append since the markers were missing).

**Why my spec missed this:** The Phase 4 prose says "issue this `ctx_execute` call verbatim" but doesn't tell the agent what to do if a prior step failed. The agent reasoned: "Step 1 failed → contract unknown → I'll do my best from training data → write report + append." That's a reasonable interpretation under ambiguity, but it defeats the entire gate system.

**Fix:** Add an explicit instruction at the top of Phase 4 (in the preflight banner): *"If any gate call errors, do NOT substitute your own logic. Print `STATUS: partial reason=<gate> <error>` and halt. The user will re-run with a fixed environment."* The current prose says "if any call errors → halt with STATUS: partial" but only inside Step 1's body — it needs to be a Phase-4-wide rule that also forbids improvisation.

## Bottom line

Phase A's core hypothesis is **validated**: hardening the prose (preflight banner + real JSON tool-call blocks + required `STATUS:` markers) converted the agent's behavior from "ignore the gates" (v0.0.8) to "try to run the gates" (v0.0.9). That is real progress and the right layer to be working at.

But the gates still don't *fire*, because two implementation details were wrong: Step 1 paths resolve against the wrong repo, and the agent improvises around failure. Both are fixable in a follow-up edit (~30 minutes of work) and a re-run. The v0.0.9 smoke test is best understood as **"Phase A round 1"** — the prompt-engineering lever works, the gate calls need path-fixing and tighter failure handling.
