## Bug-by-bug outcome

### ✅ Bug 1 (Step 1 path resolution) — fixed, with one new wrinkle

The agent issued the new `ctx_execute` form verbatim. But it hit a sandbox quirk where `ctx_execute` wraps user code in an IIFE that **overrides `fs.readFileSync`** (transcript stderr shows `f.readFileSync=function(){...}`). The first attempt errored.

**Critical positive:** the agent did *not* improvise. Instead — exactly per Bug 2's new "no improvisation" rule — it used `Bash` to locate the plugin root and re-issued the three Step 1 calls with hardcoded absolute paths. All three succeeded:
```
LOADED report.md bytes=11016
LOADED reviews-entry.json required=["ts","scope","crit","high","med","low","info","report","v","tokIn","tokOut"]
LOADED README.md bytes=4755
```

The fix works. The new wrinkle (sandbox `readFileSync` override) needs investigation but is *not* a Phase A bug — it's a context-mode MCP behavior. The agent's recovery (route around it via `Bash` + absolute paths) is precisely the kind of judgment the new preflight rule was designed to elicit, *without* the agent then bypassing the gates entirely.

### ✅ Bug 2 (agent improvisation on failure) — fixed

The agent no longer wrote ad-hoc JS to bypass Steps 4/6/7. After Step 1 succeeded, it correctly ran Step 4 (`bash validate-report.sh` → `OK`), Step 6 (`require(validate-entry.js); validateEntry(candidate)` → `OK`), and Step 7 (appended the validated entry, printed `STATUS: complete`). The methodology field in the entry no longer says *"gates did not fire"* — because they did.

### ✅ Bug 3 (Step 4 report-path prefix) — fixed

Step 4 ran cleanly: `bash "$CLAUDE_PROJECT_DIR/scripts/validate-report.sh" "<report path>"`. The script path is plugin-rooted, the report path is cwd-relative. Result: `OK`.

## The one cosmetic gap (S1 intermediate markers)

The agent printed `STATUS: complete` (twice — once at the end of its summary, once in the final `result` event) but **skipped the intermediate markers** `STATUS: gates-loaded`, `STATUS: report-ok`, `STATUS: entry-ok`. It treated them as conceptual states rather than required transcript emissions.

**Substance vs. marker:** the gates *did* fire — templates loaded (S2 ✅), both validators ran and returned `OK` (S3 ✅), entry on disk validates (S6 ✅). The intermediate markers are a smoke-test affordance, not a runtime requirement. The agent consolidated the four markers into the final `STATUS: complete` because it could see all gates had succeeded.

**Fix for a future round:** tighten Step 1/4/6 prose to make the intermediate markers literal print statements ("After all three Step 1 calls return, emit exactly the line `STATUS: gates-loaded` — this is a separate print, not a state") — same pattern that worked for Step 7's `STATUS: complete`. Low priority: the substance is right; only the test-harness grep is affected.
